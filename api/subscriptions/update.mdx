The Update endpoint lets you modify any writable field on an existing subscription. Send only the fields you want to change â€” unchanged fields are left exactly as-is. The server merges your patch over the existing record before saving, so a body with a single key is perfectly valid. If you update `lastPaymentDate` or `period`, Sub automatically recalculates `nextPaymentDate` for you.

## Endpoint

```
PATCH /api/v1/subscriptions/:id
```

## Path parameters

<ParamField path="id" type="string" required>
  UUID of the subscription to update. Must be a valid UUID v4, for example `a1b2c3d4-e5f6-7890-abcd-ef1234567890`.
</ParamField>

## Request body

All body fields are optional for a PATCH request, but you must supply **at least one**.

<ParamField body="name" type="string">
  Updated service name.
</ParamField>

<ParamField body="category" type="string">
  Updated category label.
</ParamField>

<ParamField body="amount" type="number">
  Updated cost per billing period.
</ParamField>

<ParamField body="currency" type="string">
  Updated ISO 4217 currency code. Must be one of: `CNY`, `USD`, `EUR`, `JPY`, `GBP`, `AUD`, `CAD`, `CHF`, `HKD`, `SGD`.
</ParamField>

<ParamField body="period" type="string">
  Updated billing frequency. Must be one of: `monthly`, `yearly`, `custom`. Changing this field causes `nextPaymentDate` to be recalculated automatically.
</ParamField>

<ParamField body="lastPaymentDate" type="string">
  Updated most recent payment date in `YYYY-MM-DD` format. Changing this field causes `nextPaymentDate` to be recalculated automatically.
</ParamField>

<ParamField body="customDate" type="string">
  Updated custom billing interval. Only meaningful when `period` is `custom`. If you change `period` away from `custom` without including `customDate` in the patch, Sub clears `customDate` automatically.
</ParamField>

<ParamField body="notificationEnabled" type="boolean">
  Updated notification preference for this subscription.
</ParamField>

<Warning>
  The fields `id`, `nextPaymentDate`, `createdAt`, and `updatedAt` are managed by the server. Including any of them in the request body returns a `400` error.
</Warning>

## Request example

Update only the amount for a subscription:

```bash
curl -X PATCH \
  -H "Authorization: Bearer subm_YOUR_KEY" \
  -H "Content-Type: application/json" \
  -d '{"amount":17.99}' \
  https://sub.jerrylu.app/api/v1/subscriptions/a1b2c3d4-...
```

## Response

Returns `200 OK` with an envelope containing the full updated subscription object under `data`.

<ResponseField name="data" type="object" required>
  The complete subscription object reflecting all changes.

  <Expandable title="Subscription object fields">
    <ResponseField name="id" type="string" required>
      UUID of the subscription. Unchanged.
    </ResponseField>

    <ResponseField name="name" type="string" required>
      Service name, updated if you included it in the patch.
    </ResponseField>

    <ResponseField name="category" type="string" required>
      Category label, updated if you included it in the patch.
    </ResponseField>

    <ResponseField name="amount" type="number" required>
      Cost per billing period, updated if you included it in the patch.
    </ResponseField>

    <ResponseField name="currency" type="string" required>
      ISO 4217 currency code, updated if you included it in the patch.
    </ResponseField>

    <ResponseField name="period" type="string" required>
      Billing frequency, updated if you included it in the patch.
    </ResponseField>

    <ResponseField name="lastPaymentDate" type="string" required>
      Most recent payment date in `YYYY-MM-DD` format, updated if you included it in the patch.
    </ResponseField>

    <ResponseField name="nextPaymentDate" type="string" required>
      Server-recalculated next payment date in `YYYY-MM-DD` format. Refreshed whenever `lastPaymentDate` or `period` changes.
    </ResponseField>

    <ResponseField name="customDate" type="string">
      Custom billing interval, updated if you included it in the patch. Only present in the response when `period` is `custom`; omitted otherwise.
    </ResponseField>

    <ResponseField name="notificationEnabled" type="boolean" required>
      Notification preference, updated if you included it in the patch.
    </ResponseField>

    <ResponseField name="createdAt" type="string" required>
      ISO 8601 timestamp of when the subscription was originally created. Never changes.
    </ResponseField>

    <ResponseField name="updatedAt" type="string" required>
      ISO 8601 timestamp of this update.
    </ResponseField>
  </Expandable>
</ResponseField>

<ResponseField name="requestId" type="string" required>

</ResponseField>

## Errors

| HTTP status | Error code | Description |
|---|---|---|
| `400` | `invalid_subscription_id` | The `:id` path segment is not a valid UUID. |
| `400` | `empty_patch` | The request body contained no writable fields. |
| `400` | `invalid_json` | The request body is not valid JSON or is not a JSON object. |
| `400` | `invalid_subscription` | One or more fields failed validation after merging with the existing record. |
| `400` | `invalid_subscription_field` | The body contains a field that is not writable (for example `nextPaymentDate`). |
| `404` | `subscription_not_found` | No subscription with that ID exists for your account. |
